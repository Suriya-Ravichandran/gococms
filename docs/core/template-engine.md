# Template Engine

> The GOCO CMS template package system — distributable, framework-agnostic markup + component libraries that render through ZealPHP and plug into themes, layouts, and regions without ever touching core.

**Stability:** `beta` · **Package:** `gococms/template-engine` · **Namespace:** `Goco\Template` · **Facade:** `Goco\SDK\Theme` (regions/layouts), `Goco\SDK\Widget` (component rendering)

A **template** in GOCO is a *packaged, versioned unit of presentation* — a bundle of layouts, reusable components, front-end assets, translations, and hooks that is authored against a chosen CSS framework through an **adapter** and rendered by the ZealPHP runtime. Templates are distinct from **themes**: a theme is the site-level design identity (palette, typography, brand tokens, region wiring — see [theme-engine.md](theme-engine.md)); a *template package* is a reusable, marketplace-distributable set of building blocks that themes and layouts *consume*. One theme references one or more template packages; one template package can be reused across many themes and websites.

---

## 1. Purpose

The Template Engine exists to give front-end developers a first-class, isolated way to ship presentation code as installable packages.

- **Separation of concerns.** Presentation (markup, components, CSS/JS assets, i18n strings) lives in template packages under `templates/`, cleanly separated from data (`packages/database`), behavior (`plugins/`), and identity (`themes/`).
- **Framework freedom.** A template declares which CSS framework it targets (Bootstrap 5, Tailwind, Bulma, UIkit, Foundation, Material, Pure, or Custom). A common **Adapter** contract normalizes grid, utility, and component class output so the same GOCO widget/layout API produces correct markup regardless of framework.
- **Core is immutable to template authors.** Template developers **never** edit `core/` or `packages/*`. They extend exclusively through the SDK facades, hooks, and the `templates/<slug>/` folder contract. Upgrading GOCO core never overwrites a template.
- **Render through ZealPHP.** Templates render via `App::render`, `App::renderToString`, `App::renderStream`, and `App::fragment`, inheriting per-coroutine isolation, streaming, and htmx fragment support from the runtime.
- **Distribution.** Templates are Composer packages (`gococms/template-*` or third-party) and marketplace artifacts with a `template.json` descriptor and a signed `manifest.json`.

> **Note**
> "Template package" (this document) ≠ "template file". A ZealPHP template *file* is a `.php` view rendered by `App::render`. A GOCO template *package* is a distributable bundle that *contains* many such view files plus assets, components, and metadata.

## 2. Functional Specification

### 2.1 What a template package provides

| Capability | Description |
| --- | --- |
| **Layouts** | Full-page skeletons that define named regions a theme maps into (`header`, `main`, `sidebar`, `footer`, …). |
| **Components** | Reusable, parameterized presentational partials (cards, hero, breadcrumbs, pagination) with a typed props contract. |
| **Widget renderers** | Framework-specific view files that a registered `Goco\SDK\Widget` type renders through, resolved per active template. |
| **Assets** | CSS, JS, images, fonts bundled and fingerprinted; delivered via the storage/asset pipeline behind Traefik. |
| **Translations** | Per-locale message catalogs (`translations/<locale>.json`) consumed by the `t()` helper. |
| **Adapter binding** | Declares the target CSS framework; the engine loads the matching `FrameworkAdapter`. |
| **Hooks** | `hooks/*.php` files that register `Hook::listen`/`Hook::filter` callbacks scoped to the template slug. |

### 2.2 Descriptor files

Every template ships two JSON files at its root:

- **`template.json`** — the human-authored *descriptor*: identity, framework, layouts, regions, component index, asset entry points, i18n locales, dependencies, capabilities. Edited by the developer.
- **`manifest.json`** — the *build/distribution artifact*: content hashes, resolved asset fingerprints, engine compatibility range, signature, and the frozen registry entry. Generated by `goco template:build`; never hand-edited.

### 2.3 CSS framework adapters

A template targets exactly one framework via an **adapter**. Adapters implement `Goco\Template\Adapter\FrameworkAdapter` and translate abstract layout primitives (grid, column spans, spacing, visual state) into concrete class strings, so the GOCO layout hierarchy (Section → Container → Row → Column → Widget) emits correct framework markup.

| Adapter id | Framework | Grid model | Notes |
| --- | --- | --- | --- |
| `bootstrap` | Bootstrap 5 | 12-col `.row/.col-*` | `stable` |
| `tailwind` | Tailwind CSS 3.4 | utility `grid grid-cols-*` | JIT-safe class allowlist emitted at build |
| `bulma` | Bulma | `.columns/.column` | `beta` |
| `uikit` | UIkit 3 | `uk-grid` + `uk-width-*` | `beta` |
| `foundation` | Foundation 6 | XY grid `.grid-x/.cell` | `beta` |
| `material` | Material (MDC/Material Web) | 12-col MDC layout grid | `experimental` |
| `pure` | Pure.css | `.pure-g/.pure-u-*` | `beta` |
| `custom` | Custom / none | author-supplied mapping | escape hatch; template ships its own CSS |

```php
interface FrameworkAdapter
{
    public function id(): string;                                   // 'tailwind'
    public function grid(int $columns): string;                     // container/row class
    public function column(int $span, int $of = 12, array $bp = []): string; // responsive column
    public function spacing(string $token): string;                 // 'gap-lg' -> framework class
    public function visibility(array $breakpoints): string;         // show/hide utilities
    public function stateClass(string $state): string;              // 'is-active', 'active', …
    public function assetHandles(): array;                          // required css/js handles
}
```

The **Custom** adapter is a no-op mapping (returns the author's own class strings verbatim) — used by templates that ship a bespoke CSS system and want full control.

### 2.4 Rendering flow (through ZealPHP)

1. A request resolves to a **website → theme → layout** (see [rendering-pipeline.md](../architecture/rendering-pipeline.md)).
2. The theme names a **template package** and a **layout id**; the engine loads that layout view file.
3. The engine resolves the active **adapter** and injects layout/region data.
4. The layout is rendered with `App::render('/templates/aurora/layouts/landing.php', $data)` (or streamed with `App::renderStream` for progressive HTML).
5. Region placeholders are filled by rendering assigned **widgets** through the template's widget renderer files; each `Widget::render()` output is inserted.
6. htmx-targeted regions re-render as isolated fragments via `App::fragment()` — only the changed region's HTML is returned.

## 3. Business Requirements

- **BR-1 — Reuse across sites.** A single template package must be installable on unlimited workspaces/websites without duplication; per-site config lives in `settings`, not in the package.
- **BR-2 — Framework neutrality.** The same content model must render under any supported framework by swapping adapters, enabling agencies to standardize on one framework while reusing GOCO widgets.
- **BR-3 — Zero core edits.** Template authors must accomplish 100% of customization through the folder contract, SDK, and hooks. A core upgrade must never break a compliant template.
- **BR-4 — Marketplace-ready.** Every template must be packageable as a signed, versioned artifact discoverable in the [Plugin Marketplace](../marketplace/overview.md), with SemVer and Conventional Commits.
- **BR-5 — Localizable.** Templates must externalize all user-visible strings to `translations/` and support RTL and locale fallback.
- **BR-6 — Performant delivery.** Assets must be fingerprinted, cacheable, and served with critical-CSS inlining and long-lived cache headers behind Traefik.
- **BR-7 — Multi-tenant safe.** Rendering must honor `workspace_id`/`website_id` isolation; a template installed for one tenant must not leak state to another (see [multi-tenancy.md](../architecture/multi-tenancy.md)).

## 4. User Stories

- **As a template developer**, I scaffold a new template with `goco make:template aurora`, choose `tailwind`, and get a working folder with `template.json`, layouts, and a demo component — so I start productive in one command.
- **As a template developer**, I add a `hero` component with a typed props schema and preview it with `goco template:preview aurora hero` — so I validate markup without spinning up a full site.
- **As a designer**, I switch a template's adapter from `bootstrap` to `tailwind` in `template.json`, rebuild, and see the same layouts re-emit correct utility classes — so I migrate frameworks without rewriting content.
- **As a website-admin**, I install a marketplace template, assign it to a theme's layout, and map widgets into its regions from the Page Builder — so I restyle a site without code (see [page-builder.md](page-builder.md)).
- **As a translator**, I add `translations/ta.json`, and the site renders Tamil strings with English fallback — so I localize without touching markup.
- **As a platform operator**, I pin a template version per website and roll it back after a bad release — so production stays stable.

## 5. Data Model (MongoDB collections & indexes)

Template packages live on disk under `templates/`, but their *installed, resolved* state is registered in MongoDB so tenants can enable, version-pin, and configure them. The engine owns the `templates` collection — a **module-owned collection** that extends the core [Data Model](../architecture/data-model.md) catalog — and reuses the shared `settings` collection.

### 5.1 `templates` collection

```javascript
{
  _id: ObjectId,
  workspace_id: ObjectId,          // tenant scope (null = platform-global/marketplace-cached)
  website_id: ObjectId | null,     // null = available to all websites in workspace
  slug: "aurora",
  name: "Aurora",
  package_version: "1.4.0",        // installed version (SemVer)
  kind: "template",                // distinguishes from theme rows sharing infra
  framework: { adapter: "tailwind", version: "3.4" },
  layouts: [ { id: "landing", regions: ["header","hero","main","cta","footer"] } ],
  components: [ "hero", "feature-grid", "card", "pagination" ],
  locales: ["en","es","de","ta"],
  assets: { css: ["/assets/t/aurora/app.4f2a1c.css"], js: ["/assets/t/aurora/app.9c02de.js"] },
  manifest_hash: "sha256-7f3b…",
  source: { type: "marketplace", package: "northwind/aurora", signed: true },
  status: "active",                // active | disabled | installing | failed
  installed_by: ObjectId,
  // envelope fields
  created_at: ISODate, updated_at: ISODate, deleted_at: null,
  version: 3, created_by: ObjectId, updated_by: ObjectId
}
```

### 5.2 Per-site template configuration

Site-level overrides (chosen layout per page type, adapter options, feature flags) are stored in the `settings` collection under a `template:<slug>` namespace, keyed by `(workspace_id, website_id)` — never inside the package.

## 7. API Design

### 7.1 File-based rendering endpoints

Template previews and asset delivery use ZealPHP file-based REST and routing. Asset URLs are stable, fingerprinted, and served with immutable cache headers behind Traefik.

| Method | Route | Purpose |
| --- | --- | --- |
| `GET` | `/assets/t/{slug}/{file}` | Serve a fingerprinted template asset (immutable cache). |
| `GET` | `/api/templates` | List installed templates for the current tenant (auto-JSON). |
| `GET` | `/api/templates/{slug}` | Descriptor + resolved manifest for one template. |
| `GET` | `/api/templates/{slug}/preview/{component}` | Server-render a component with sample props (admin, `templates.render`). |
| `POST` | `/api/templates/{slug}/install` | Install/register a template for a tenant (`themes.manage`). |
| `PATCH` | `/api/templates/{slug}` | Pin version / set adapter options (`themes.manage`). |
| `DELETE` | `/api/templates/{slug}` | Soft-disable a template for the tenant. |

### 7.2 Runtime rendering API (ZealPHP)

```php
use ZealPHP\App;

// Full layout render
$html = App::renderToString('/templates/aurora/layouts/landing.php', [
    'regions' => $resolvedRegions,
    'adapter' => $adapter,
    't'       => $translator,   // locale-bound callable
]);

// Progressive streaming (Generator -> chunked HTTP)
$app->route('/p/{path}', function ($path, $request, $response) {
    return App::renderStream('/templates/aurora/layouts/default.php', $ctx);
});

// htmx region fragment — only re-render one region
$app->route('/fragment/feature-grid', function ($request) {
    return App::fragment('/templates/aurora/components/feature-grid.php', [
        'items' => $items, 'adapter' => $adapter,
    ]);
});
```

### 7.3 Component rendering via the Widget facade

A template component is surfaced to the page builder as a widget type; the renderer file the engine resolves is the active template's version of that type.

```php
use Goco\SDK\Widget;

// Registered once; renderer resolved per active template
Widget::register('hero-banner', [
    'label'    => 'Hero Banner',
    'category' => 'template',
    'schema'   => 'components/hero.schema.json',
    'render'   => fn(array $props, $ctx) =>
        App::renderToString($ctx->template()->view('widgets/hero-banner.php'), $props),
]);

$markup  = Widget::render('hero-banner', ['title' => 'Ship faster'], $ctx);
$preview = Widget::preview('hero-banner', ['title' => 'Sample']);
$schema  = Widget::properties('hero-banner'); // PropertySchema
```

## 8. Services

The engine is implemented as cohesive services in `packages/template-engine` (namespace `Goco\Template`), resolved from the service container (see [service-container.md](../architecture/service-container.md)).

| Service | Responsibility |
| --- | --- |
| `TemplateRegistry` | Discovers `templates/*`, parses `template.json`, syncs the `templates` collection, resolves the active template per (workspace, website). |
| `TemplateResolver` | Given theme + layout id, resolves the concrete layout view file, regions, and component map. |
| `AdapterFactory` | Loads the `FrameworkAdapter` for a template's declared framework; caches per worker. |
| `ComponentRenderer` | Renders components/widgets through ZealPHP with prop-schema validation. |
| `AssetPipeline` | Builds, fingerprints, and publishes `assets/*` to the storage driver; emits immutable URLs. |
| `TranslationLoader` | Loads `translations/<locale>.json`, provides the `t()` callable with fallback chain. |
| `ManifestBuilder` | Produces/validates `manifest.json`, computes integrity hashes, verifies signatures. |
| `TemplateInstaller` | Installs/upgrades/rolls back templates for a tenant inside a MongoDB transaction. |

Services are registered as singletons at boot; per-request state (active template, locale) flows through `\ZealPHP\G` / `RequestContext`. Resolved adapters and parsed descriptors are cached in a `\ZealPHP\Store` table so all OpenSwoole workers share a warm registry; cache invalidation on template install/upgrade is broadcast via `Store::publish` + `App::subscribe`.

## 9. Events

Emitted through `Goco\SDK\Hook` (`Hook::dispatch`, alias `do`). Actions follow `subject.verb[.tense]`.

| Event | When | Args |
| --- | --- | --- |
| `template.registered` | A template is discovered and synced to the registry | `$slug, $descriptor` |
| `template.installing` | Install transaction begins | `$slug, $version, $ctx` |
| `template.installed` | Install committed | `$slug, $version` |
| `template.activated` | Template becomes active for a website | `$slug, $websiteId` |
| `template.upgraded` | Version changed | `$slug, $from, $to` |
| `template.rolled_back` | Reverted to prior version | `$slug, $to` |
| `template.disabled` | Soft-disabled for a tenant | `$slug, $websiteId` |
| `template.render.before` | Before a layout renders | `$slug, $layout, $ctx` |
| `template.render.after` | After a layout renders | `$slug, $layout, $html` |
| `component.render.before` | Before a component/widget renders | `$type, $props, $ctx` |
| `component.render.after` | After a component/widget renders | `$type, $html` |
| `template.assets.built` | Asset pipeline finished a build | `$slug, $manifest` |

```php
use Goco\SDK\Hook;

Hook::listen('template.activated', function (string $slug, string $websiteId) {
    // warm the fragment cache for the new template
}, priority: 20);
```

## 10. Hooks

Filters (`subject.noun`, applied with `Hook::apply`, alias) let templates and plugins reshape rendering without editing core. Templates register their own hooks from `hooks/*.php`, namespaced by slug.

| Filter | Value | Purpose |
| --- | --- | --- |
| `template.layout` | `string $layoutId` | Override which layout renders for a page. |
| `template.regions` | `array` | Add/remove/reorder region assignments. |
| `template.class` | `string` | Post-process an adapter-emitted class string. |
| `template.adapter` | `FrameworkAdapter` | Swap or wrap the active adapter. |
| `widget.output` | `string $html` | Mutate a rendered widget's HTML. |
| `response.headers` | `array` | Add asset cache / security headers. |
| `menu.items` | `array` | Reshape navigation before a nav component renders. |

```php
// templates/aurora/hooks/filters.php
use Goco\SDK\Hook;

Hook::filter('aurora.template.class', function (string $classes, string $token) {
    return $token === 'gap-lg' ? "$classes gap-8" : $classes;   // adapter tweak
});

Hook::filter('widget.output', function (string $html) {
    return str_replace('<img ', '<img loading="lazy" ', $html);
}, priority: 5);
```

> **Tip**
> Prefix every template-defined hook with the template slug (`aurora.*`) so multiple installed templates never collide. See the [Hook SDK](../sdk/hook-sdk.md).

## 11. UI Architecture

- **Page Builder integration.** Template components register as widget types with a category of `template`, so they appear in the [Page Builder](page-builder.md) palette. Editors drag components into layout regions; props are edited through the `PropertySchema` form.
- **Region mapping.** The [Theme Engine](theme-engine.md) binds a theme's regions to a template layout's named regions. The admin UI shows a region map (theme region → template region) with validation for unmapped required regions.
- **Live preview.** `goco template:preview` and the `/api/templates/{slug}/preview/{component}` endpoint server-render components in isolation, streamed to the admin preview pane via SSE.
- **Adapter awareness.** The builder reads the active adapter so column controls emit the correct grid semantics (e.g., 12-col vs. utility grid) and the canvas loads the template's `assets/css/app.css`.
- **Layout hierarchy.** Rendering honors the canonical hierarchy Workspace → Website → Theme → Layout → Section → Container → Row → Column → Widget; the template supplies the Layout and the visual vocabulary for Section→Widget.

## 12. Security Model

- **No core access.** Templates cannot write outside `templates/<slug>/` and cannot autoload code outside their namespace; the loader restricts `require` to the package root.
- **Signature verification.** Marketplace templates ship an Ed25519 signature over `manifest.json` integrity hashes; `TemplateInstaller` refuses to install a template whose content hash mismatches the signed manifest (BR-4).
- **Output escaping.** All component props are escaped by default in views; the `t()` translator returns escaped strings unless `raw()` is explicitly used. `widget.output` filters run after a sanitization pass.
- **Capability gating.** Install/upgrade requires `themes.manage`; server-side previews require `templates.render`. Enforced via RBAC + optional ABAC (see [permission-system.md](../architecture/permission-system.md)).
- **CSRF & headers.** Preview/install mutations pass through the ZealPHP `Csrf` middleware; asset responses set immutable cache and security headers, hardened further by Traefik middleware (see [traefik.md](../deployment/traefik.md)).
- **Tenant isolation.** Every registry query is scoped by `workspace_id`/`website_id`; the `\ZealPHP\Store` cache is keyed by tenant to prevent cross-tenant template bleed.
- **No arbitrary PHP execution from data.** Template layouts are static files shipped in the package; user content never becomes an executed template path.

See the platform-wide [Security Model](../security/security-model.md).

## 13. Performance Strategy

- **Warm registry in shared memory.** Parsed descriptors and resolved adapters live in a cross-worker `\ZealPHP\Store` (OpenSwoole\Table), avoiding per-request JSON parsing; invalidated via `Store::publish`.
- **Coroutine rendering.** Layouts render inside coroutines; independent regions render concurrently with `go()` + `\OpenSwoole\Coroutine\Channel`, then compose.
- **Streaming + critical CSS.** `App::renderStream` yields the document head (with inlined `css/critical.css`) immediately, then streams region chunks with `co::sleep()` cooperative yields for time-to-first-byte wins.
- **htmx fragments.** `App::fragment()` re-renders only a changed region, cutting payload for interactive updates.
- **Immutable, fingerprinted assets.** `AssetPipeline` hashes assets; URLs are content-addressed and served with `Cache-Control: public, max-age=31536000, immutable` behind Traefik with HTTP/3.
- **Fragment cache.** Rendered components are cached in Redis keyed by `(template, component, props-hash, locale)`; invalidated on `template.upgraded` and content changes (see [caching-and-queue.md](../architecture/caching-and-queue.md)).

## 14. Testing Strategy

- **Descriptor validation.** Schema-validate every `template.json` against the engine's JSON Schema in CI (`goco template:lint`).
- **Adapter conformance suite.** Each `FrameworkAdapter` runs a shared golden-output test: the same primitive calls must produce framework-correct classes; snapshots per framework.
- **Component render tests.** Render each component with schema-valid and edge-case props; assert escaped output and no undefined-prop warnings.
- **Visual/regression.** Headless render of preview endpoints; DOM snapshot diffs gate merges.
- **i18n coverage.** Assert every default-locale key exists in each declared locale or has a documented fallback.
- **Integrity/signature.** Verify `goco template:build` output re-verifies against `manifest.json` and marketplace signature.
- **Multi-tenant isolation.** Install the same template for two workspaces and assert no cache/state bleed.

Follows the org [Testing Strategy](../community/testing-strategy.md) and [Coding Standards](../community/coding-standards.md).

## 15. Extension Points

- **Custom adapters.** Implement `Goco\Template\Adapter\FrameworkAdapter` and register it; a template can then declare `"adapter": "my-framework"`.
- **Component library.** Ship additional components under `components/` with prop schemas; expose them as widgets via `Widget::register`.
- **Hooks.** Use `hooks/register.php` for `Hook::listen` and `hooks/filters.php` for `Hook::filter` to alter layout resolution, classes, and output.
- **Asset transformers.** Register an `AssetPipeline` transformer (e.g., a Tailwind JIT purge step, PostCSS chain) via the plugin engine.
- **Translation providers.** Swap `TranslationLoader` for a remote/AI translation source through the container.
- **Layout/region contributions.** Plugins may contribute regions consumed by templates via the `template.regions` filter.

The full authoring walkthrough lives in the [Template Guide](../guides/template-guide.md).

## 16. Upgrade Strategy

- **SemVer + engine range.** `template.json.engine.goco` declares the compatible core range; `TemplateInstaller` refuses incompatible installs.
- **Version pinning per website.** Each tenant pins a template version in the `templates` collection; upgrades are explicit.
- **Transactional upgrade.** Upgrades run inside a MongoDB multi-document transaction updating the registry row and `settings`; failure rolls back atomically, emitting `template.rolled_back`.
- **Migrations.** A template may ship `hooks/upgrade/<from>-<to>.php` migration callbacks invoked on `template.upgrading`; used to remap renamed regions/components.
- **Deprecation.** Removed components/layouts are marked `deprecated` in `template.json` for one minor cycle before removal; the linter warns consumers.
- **Zero-downtime assets.** New fingerprinted assets publish before the registry flips the active version, so in-flight requests keep serving old assets.

## 17. Future Roadmap

- **Theme UI editor round-trip** — visual edits persisting back to component prop defaults.
- **AI-assisted templating** — generate components and adapter mappings from a prompt via the [AI Platform](ai-platform.md).
- **Design-token bridge** — shared token layer (colors, spacing, type) that feeds every adapter for cross-framework parity.
- **Partial hydration / islands** — ship interactive components as hydration islands with per-island JS budgets.
- **Container queries first-class** — adapter primitives for container-query responsive layouts.
- **Template composition** — declare a template that extends/overrides another (`extends` in `template.json`).
- **Marketplace analytics** — install/usage telemetry surfaced to authors (see [marketplace/overview.md](../marketplace/overview.md)).

See the platform [Roadmap](../roadmap.md).
